## Технічна схема вузлів

```
IMAP Trigger (тема «довідка»)
  → Find PDF attachment (перший mimeType=application/pdf; attachment_0 може бути логотипом!)
  → Extract invoice text (pdf-parse)
  → Extract clean email text
  → Build Prompt (Code node) ← будує prompt для Claude
  → Call Claude API (HTTP Request, Anthropic API)
  → Prepare document data (Code node) ← парсить JSON-відповідь, рахує суму
  → Filter border services ← фільтрує послуги за кордоном
  → Copy Google Doc template
  → Fill placeholders ({{date}}, {{main_text}}, {{insurance_phrase}})
  → Export DOCX
  → Rename file (Довідка_<компанія>_<маршрут>_<дата>.docx)
  → Send Email Reply (SMTP, docs@unitex.od.ua, In-Reply-To)
```

**Конфіги:**
```json

```
Ключ у `~/.n8n_env` і `~/.claude/projects/-Users-irina/.env` (НЕ видаляти без команди користувача).

## Правила відбору послуг «до кордону»

### ВКЛЮЧАТИ якщо хоча б одна умова:
1. Маркер «за межами державного/митного кордону України» або «до державного кордону України»
2. Авіафрахт або морський фрахт (навіть без маркера)
3. «міжнародне перевезення/авіаперевезення/автоперевезення» І маршрут закінчується за кордоном або у пункті перетину кордону:
   - м/п: Мостиська, Угрінів, Устилуг, Рава-Руська, Ягодин, Краківець
4. Маршрут містить ТІЛЬКИ іноземні міста

### ВИКЛЮЧАТИ:
- Маршрут закінчується у внутрішньому місті України (навіть якщо починається з-за кордону).
  Приклад: «Варшава → м. Солоницівка» → **виключити**
- «на території України» без маршруту за кордон
- Агентська/експедиторська винагорода без маршруту
- ❌ НЕ виключати страхування (воно йде в `insurance_amount` і додається до суми)

### Страхування — особливе правило:
- Страхування **завжди включається до `border_total`** (не окремо)
- `border_total = border_total_raw + insurance_amount`
- «страхування вантажу» додається до переліку послуг у тексті
- `insurance_phrase = ''` якщо страхування є; `"Страхування вантажу не проводилось."` якщо немає

## Перевізні документи (в дужках після «вантажу»)

| Тип | Формат |
|-----|--------|
| AWB/MAWB | авіанакладна XXX |
| Коносамент | коносамент XXX |
| CMR | **з тіла листа** (`email_text`), НЕ з PDF → «CMR XXX від DD.MM.YYYY» |
| Контейнер | контейнер XXXX1234567 |

> ⚠️ CMR завжди читати з `email_text` (з вузла «Extract clean email text»), а не з PDF.
> Авіафрахтові рахунки (наприклад, №0000023) CMR не мають — там лише AWB.

## Ключові рішення, прийняті під час розробки

1. **AI-нода замість regex** — Claude читає текст PDF і сам витягує маршрут, суму, тип послуг. Це надійніше за складний Code node з регулярними виразами.
2. **PDF може бути не першим вкладенням** — нода «Find PDF attachment» шукає перший файл з `mimeType=application/pdf`, а не `attachment_0`.
3. **Універсальність — критично важлива** — workflow не прив'язаний до конкретного рахунку або номера. Будь-який PDF-рахунок від Юнітексу має оброблятись коректно.
4. **Сума рахується Claude-ом, не regex** — після того, як ручний парсинг давав помилки, перейшли на LLM-витяг суми.
5. **Страхування завжди в border_total** — рішення прийнято в останньому повідомленні основної сесії.
