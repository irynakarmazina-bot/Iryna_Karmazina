## Ключові рішення (і чому)

1. **Матч-ключ запису — колонка «Угода», а не BL/Контейнер (проблема з 1.2).** Замість двох гілок запису або нової службової колонки використав уже наявну (SHEET-STRUCTURE.md, колонка A) «Угода» — унікальний номер угоди Експедитора, завжди заповнений, не залежить від того, чим трекається рядок цього прогону. Це найпростіший і найнадійніший варіант: одна гілка запису для всіх сценаріїв (BL/контейнер/mark-only), без ризику appendOrUpdate матчити «сусідній схожий» рядок. Обґрунтовано коментарем у `vidbir.code.js`.

2. **«Останнє записане скриптом значення» (потрібне для guard 1.5) зберігається в самому «Журналі автооновлень», а не в нових прихованих колонках CRM.** Журнал — append-only історія «Ключ+Колонка→Старе/Нове», тож останній запис по (Ключ,Колонка,Джерело) вже є тим значенням, яке потрібне guard. Це уникає дублювання даних (ще одна копія в CRM) і додаткових колонок. Ціна рішення: після автопереходу «контейнер→BL» ключ у журналі змінюється, і guard тимчасово «не пам'ятає» історію під новим ключем на перший прогін після переходу — задокументовано в коментарі коду як прийнятний компроміс.

3. **HTTP-запит до Maersk API виконує сам код-вузол «Розбір» (`this.helpers.httpRequest`), а не окремий HTTP Request-вузол.** Причина: потрібен точний retry з різними паузами (2с/4с/8с) і чітке розрізнення 429 (повторити)/401 (зупинити все)/404/порожній events (позначити рядок)/будь-що інше (позначити як формат) — вбудований retry n8n-вузла має лише СТАЛУ паузу і не розрізняє коди статусів для різної поведінки. Мінус рішення: код-вузол більший і складніший для читання; я максимально прокоментував логіку.

4. **Гранулярність guard — по полю** (Статус/ETA план/ETA факт/Судно окремо), реалізовано через `guardOk()`: поле просто не потрапляє в об'єкт запису `out`, якщо є конфлікт — оскільки `appendOrUpdate` з `autoMapInputData` чіпає лише ті колонки, які присутні в об'єкті. Це і є механізм «не зачепити решту полів» з задачі 1.5.

5. **Гілка «mark» (вкажіть лінію / не трекається) обходить весь HTTP/guard-пайплайн** і йде напряму в «Запис у CRM» — вона взагалі не потребує HTTP-запиту й guard тут не застосовний («Ключ трекінгу» — суто скриптове поле, не з переліку 4 guard-полів).

## Як кожен критерій 1.1–1.5 виконується (коротко)

- **1.1**: retry 429 з backoff 2/4/8с, ≤3 повтори — в `rozbir.code.js` (цикл `while(attempt<maxAttempts)`). 401 на весь запуск подій → `stop-401` маркер → `Запис логу (стоп)` → `Стоп: 401 (events)` (execution завершується статусом помилки, попередні рядки цього прогону очищені з результату). Токен-рівень 401 — окрема гілка `Токен дійсний?`→`Стоп: недійсний токен`. BL не знайдено/порожній events → `markRow(...,'BL не знайдено')`, лише цей рядок. Несподіваний формат/timeout → `try/catch` + перевірка `Array.isArray(body.events)` → `markRow(...,'формат/timeout')`. Жоден шлях не зациклюється нескінченно (цикл обмежений `maxAttempts`).
- **1.2**: повністю в `vidbir.code.js` (визначення ключа/маршруту) + `rozbir.code.js` (побудова URL, запис «Ключ трекінгу», автоперехід). Матч-ключ запису — «Угода» (рішення 1 вище).
- **1.3**: таблиця мапінгу код×мода в `rozbir.code.js` + `STATUS-MODEL.md`; заморозка — окрема перевірка `frozen` перед розрахунком статусу; «дірки» — гілка `else` з Нагадування+журнал.
- **1.4**: кожна зміна (Контейнер/Судно/ETA план/ETA факт/Статус/Ключ трекінгу) штовхає окремий `{_kind:'log', ...}` item лише коли значення дійсно змінюється; `SOURCE` — одна константа; «останнє авто-значення» зберігається імпліцитно в самому журналі (рішення 2 вище).
- **1.5**: `guardOk()` застосований до Статусу/ETA(план)/ETA(факт)/Судна; «Контейнер» — fill-if-empty; поле, що НЕ пройшло guard, просто не потрапляє в об'єкт запису (guard по полю, решта полів пишуться штатно); конфлікт → текст у «Нагадування»; TOCTOU-обмеження задокументоване коментарем у `rozbir.code.js`.
