# ПЛАН РЕАЛІЗАЦІЇ (фінальний) — Таблиця диспетчеризації (міні-CRM)

Версія після проходження через критика і автора тестів. Критерії приймання кожної задачі — у ACCEPTANCE-CRITERIA.md (валідатор Sonnet звіряє буквально).

## Принципи (діють у КОЖНІЙ задачі)
- Кожна задача самодостатня: вхід / результат / критерій готовності. Явні залежності позначені «ЗАЛЕЖИТЬ ВІД».
- Один господар колонки. **Виняток задекларовано:** колонка «Контейнер» — скрипт заповнює ТІЛЬКИ якщо порожня; якщо людина вписала — не чіпає. Це не «дуальний господар», а правило «fill-if-empty».
- Скрипт ніколи не перетирає людські колонки; при розбіжності трекованого поля пише в «Нагадування», не в саме поле.
- Кожне авто-оновлення логується (журнал: що/коли/звідки/старе→нове).
- Обережний темп ліній (робочі акаунти, блокування недопустиме).
- Реалізація — Sonnet-виконавці; після кожної задачі — Sonnet-валідатор за критеріями приймання.
- n8n-деплой: користувачка імпортує JSON вручну АБО через VPS (прямого API-доступу в сесії поки нема).

### Задача 1.1+1.2 — Maersk: видимість збоїв + ключ трекінгу (ОБ'ЄДНАНО; правка критика п.29)
ПРИЧИНА об'єднання: обидві змінюють match-ключ запису — окремо їх приймати не можна, друга переробить першу.
ЗРОБИТИ (видимість збоїв): 429 → повтор з backoff (≤3) у межах прогону, після вичерпання → «потребує перевірки» + лог; 401 на весь батч → зупинити прогін з явним записом у лог (не тихий успіх); BL не знайдено → позначка тільки цього рядка + лог; несподіваний формат/timeout → «потребує перевірки» + лог. Ніде не зависати, не циклити нескінченно.
ЗРОБИТИ (ключ трекінгу): BL(9 цифр) → по BL; нема BL, є контейнер+Лінія=Maersk → по контейнеру, «Ключ трекінгу»=«контейнер (тимчасово)», перевірити, що Maersk API приймає equipmentReference; запис матчиться по відповідному ключу; з'явився BL → перехід на BL + лог «контейнер→BL»; нема обох → «не трекається»; контейнер без лінії → «вкажіть лінію», без запиту.
ГОТОВО: критерії 1.1 і 1.2 з ACCEPTANCE-CRITERIA.md.

### Задача 1.3 — Maersk: нова 8-модель статусів + ОНОВЛЕННЯ КОДУ ДОШОК (правка критика п.25)
ЗРОБИТИ: перемапити події Maersk → 8 нових статусів; оновити STATUS-MODEL.md; заморозка імпорт=«Вантаж доставлено», експорт=«Вивантажений в порту прибуття». **ОБОВ'ЯЗКОВО:** оновити масиви blue/purple і функцію pill() у dashboard.code.js І portal.code.js під нову модель (інакше кольори статусів мовчки зламаються — старі назви «Зданий в порт»/«Прибув в сухій порт» зашиті в коді).
ГОТОВО: критерії 1.3 + жоден старий статус не лишився в коді дошок; кольори працюють для всіх 8.

### Задача 1.4 — Журнал усіх авто-змін (передумова для 1.5)
ЗРОБИТИ: аркуш «Журнал автооновлень» (дата/час, ключ, колонка, старе→нове, джерело); логувати Статус/ETA/Судно/Контейнер, не лише ETA; код параметризований по джерелу (готовий для MSC/CMA); зберігати «останнє записане скриптом значення» по рядку (потрібно для guard 1.5).
ГОТОВО: критерії 1.4.

### Задача 1.5 — Guard: захист ручних правок (АЛГОРИТМ; правка критика п.6-8, п.30)
ЗАЛЕЖИТЬ ВІД: 1.4 (журнал «останнє авто-значення»).
АЛГОРИТМ (замість розмитого «схоже на ручну правку»): порівнювати ПОТОЧНЕ значення в таблиці з «останнім записаним скриптом» (з 1.4), а не з читанням на початку прогону. Якщо поточне ≠ останнє-авто → людина редагувала → не перетирати, писати розбіжність у «Нагадування». Гранулярність — ПО ПОЛЮ: guard виключає лише конфліктне поле з набору запису, решту полів рядка оновлює.
ЧЕСНЕ ОБМЕЖЕННЯ (задокументувати): guard захищає правки, зроблені МІЖ прогонами. Правку, зроблену всередині одного прогону (у вікні читання→запис, хвилини), guard цього разу може не побачити — це прийнятно за рідкості (1 прогін/добу, вузьке вікно).
ГОТОВО: критерії 1.5, включно з обов'язковим тестом «ручна правка не перетерта» для Статус/ETA/Судно/Контейнер.
