### Задача 1.1 — Maersk: видимість збоїв
- Підставити (замокати HTTP-відповідь вузла «Maersk API») відповідь **429** на один із запитів батчу → Воркфлоу повторює саме цей запит з backoff у межах поточного прогону (не одразу позначає рядок «потребує перевірки»); кількість повторів обмежена (≤3), після вичерпання лімітів — рядок отримує позначку в «Нагадування» + запис у лог.
- Підставити відповідь **401** на ВЕСЬ батч (протухлий токен) → Прогін зупиняється явно (execution завершується зі статусом помилки/зупинки), у лог-аркуш падає один запис «401 — недійсний токен, прогін зупинено, <дата>»; жоден рядок таблиці НЕ отримує «оновлено» чи «успіх», нема тихого проходження без даних.
- Підставити відповідь «BL не знайдено» (порожній `events`/404) для одного BL при нормальних інших → Тільки цей рядок отримує в колонці «Нагадування» позначку «потребує перевірки: BL не знайдено <дата>» + рядок (BL, причина, дата) у лог-аркуш; решта батчу обробляється штатно (перевірити на сусідніх тестових рядках).
- Підставити відповідь у несподіваному форматі (HTML замість JSON / відсутнє поле `events` / timeout) → Рядок отримує «потребує перевірки: формат/timeout <дата>» + запис у лог; воркфлоу завершує execution без зависання (перевірити, що execution не лишається в стані running).
- Прогнати всі 4 сценарії підряд в одному тестовому прогоні → Жоден не викликає нескінченного циклу повторів (перевірити ліміт повторів у коді retry/backoff вузла), воркфлоу завершується коректно у кожному випадку.

### Задача 1.2 — ключ трекінгу (BL / контейнер-fallback / автоперехід)
- Тестовий рядок: BL=9 цифр, контейнер порожній, Лінія=Maersk → Ключ трекінгу=BL; запит до Maersk іде по `carrierBookingReference`; запис результату потрапляє саме в цей рядок (перевірити matching по BL, не по сусідньому схожому значенню).
- Тестовий рядок: BL порожній, контейнер заповнений, Лінія=Maersk → «Ключ трекінгу»=«контейнер (тимчасово)»; запит іде по `equipmentReference` (підтвердити документацією/тестовим викликом Maersk API, що параметр приймається — інакше критерій не зданий); запис матчиться по контейнеру.
- Тестовий рядок: контейнер заповнений, Лінія порожня → Рядок отримує позначку «вкажіть лінію»; жодного HTTP-запиту до Maersk API для цього рядка не робиться (перевірити в логах вузла відбору, що рядок відфільтрований до запиту).
- Тестовий рядок: BL і контейнер обидва порожні → «Ключ трекінгу»=«не трекається»; рядок пропускається без помилки воркфлоу.
- Сценарій автопереходу: рядок раніше трекався по контейнеру (BL порожній) → людина вписує BL у цей рядок → наступний прогін → Ключ трекінгу переключається на BL; у лозі/«Зміни ETA (історія)» з'являється запис «контейнер → BL, <дата>».

### Задача 1.3 — перемапінг статусів у нову 8-модель
- Прочитати оновлений `docs/STATUS-MODEL.md` → Містить лише узгоджену послідовність (Букінг → Стафіровка → В порту відправлення → Завантажений на судно → В морі → Вивантажений в порту прибуття → [імпорт] Завантажений на авто/потяг → Вантаж доставлено); статуси «На кордоні», «Зданий в порт», «Прибув в сухій порт», «Доставка отримувачу» відсутні як окремі канонічні стадії.
- Пройти по кожній комбінації код-події×режим-транспорту (LOAD/DEPA/ARRI/DISC/GTOT/GTIN × VESSEL/RAIL/TRUCK) в оновленому `rozbir.code.js` → Кожна комбінація мапиться лише на статус із нового 8-переліку (жодного зі старих 4 прибраних статусів на виході).
- Тестовий рядок Напрямок=Імпорт, поточний статус = «Вантаж доставлено», подати нову подію Maersk → Статус НЕ змінюється (заморозка тримається).
- Тестовий рядок Напрямок=Експорт, поточний статус = «Вивантажений в порту прибуття» (експорт-фінал), подати нову подію → Статус НЕ відкочується і не змінюється (фінал заморожений так само, як імпортний фінал).
- Перевірити відсутність «дірок»: жодна очікувана комбінація код+режим не залишає поле статусу порожнім/незміненим мовчки без відповідного запису в «Нагадування» чи лог.

### Задача 1.4 — логування всіх авто-змін
- Перевірити наявність окремого аркуша «Журнал автооновлень» з колонками дата/час, ключ (BL/контейнер), колонка, старе→нове, джерело.
- Тестовий прогін зі зміною Статусу одного рядка → У «Журналі автооновлень» з'являється новий рядок з точними полями (старе значення, нове значення, колонка=«Статус», джерело=«Maersk», дата/час заповнені).
- Аналогічний тест для ETA (план), Судна, Контейнера → кожна зміна логується окремим рядком журналу.
- Тестовий прогін, де нове значення дорівнює старому (без реальної зміни) → Новий рядок у журналі НЕ створюється (немає дублюючого спаму).
- Перевірити код запису в журнал параметризований по джерелу (не hardcoded «Maersk») → готовий до підключення MSC/CMA без переробки.

### Задача 1.5 — захист колонок людини (guard)
- **[ОБОВ'ЯЗКОВИЙ ТЕСТ]** Вручну вписати в «Статус» значення, що відрізняється від того, яке видасть скрипт на цьому прогоні → прогнати воркфлоу → Комірка «Статус» лишається ручним значенням (не перезаписана); у «Нагадування» з'явився запис розбіжності (старе-скриптове vs ручне, дата).
- Той самий тест повторити для ETA (план/факт), Судна, Контейнера окремо → у кожному випадку ручне значення збережене + запис розбіжності.
- Переглянути список полів, які пише кожен трекінг-воркфлоу (по коду вузлів запису) → Жодна людська колонка (коментарі, дати-факти L/M/N/O/T/U/V/W/X/Y/Z/AA/AB/AM–AY тощо) не з'являється серед полів запису в жодній гілці коду.
- Тестовий рядок, де в трекованому полі стоїть попереднє авто-значення (не відредаговане людиною) → скрипт оновлює його новим значенням як завжди (guard не блокує штатне оновлення).
- Прогнати той самий воркфлоу двічі поспіль без жодної ручної правки між прогонами → Другий прогін НЕ генерує хибний запис «розбіжність» (guard не спрацьовує на власні попередні авто-значення).

### Задача 1.6 — MSC: воркфлоу публічного трекінгу
- Прогнати на N=3–5 тестових BL/контейнерах MSC → Дата прибуття/судно/статус записані у відповідні колонки, за тими ж правилами ключа/guard/логу, що й Maersk (1.2/1.4/1.5).
- Перевірити конфігурацію: розклад = 1 прогін/добу; між HTTP-запитами є пауза (delay-вузол); обробка послідовна, не паралельна (без Promise.all/паралельного SplitInBatches).
- Змоделювати недоступність сайту MSC (timeout/5xx/HTML замість очікуваних даних) → рядки отримують «потребує перевірки», запис у лог, execution завершується без зависання.
- Змоделювати помилку на 2-му з 5 тестових запитів → воркфлоу зупиняється після першої помилки й НЕ продовжує запити 3–5 у цьому ж прогоні (стоп після першої помилки).
- Контрольна звірка N≤5 MSC-контейнерів вручну із сайтом лінії (див. наскрізний тест 5).

### Задача 1.7 — CMA: публічний трекінг + напівручний режим
- Спроба авточитання на тестовому BL CMA, якщо DataDome не блокує → дані записані за правилами ключа/guard/логу.
- Змоделювати блокування DataDome (challenge-сторінка/CAPTCHA/незвичний HTML) → воркфлоу НЕ повторює спроби нескінченно (одна спроба чи обмежена кількість) і переходить у напівручний режим: рядок з'являється в аркуші/вигляді «Перевірити сьогодні (CMA)».
- Перевірити напівручний аркуш → для кожного BL є готове клікабельне посилання на публічний трекінг CMA з підставленим значенням BL (не голий текст BL).
- Позначити рядок у напівручному аркуші як «перевірено» → повторний прогін НЕ показує цей рядок знову як «потребує перевірки сьогодні» (розрізняє оброблені/необроблені).
- Перевірити, що механізм напівручного режиму спільний (той самий код/аркуш) для MSC, якщо MSC теж заблокується — не окрема реалізація.
- Перевірити відсутність повторного стуку: після невдалої спроби в межах прогону CMA-воркфлоу НЕ робить другого запиту до того самого BL.

### Задача 1.8 — Безпека
- Відкрити URL дашборда без секретного параметра/без Basic Auth заголовка → доступ відхилено (401/403), HTML з даними всіх відправлень НЕ віддається.
- Відкрити той самий URL з правильним секретом/логіном-паролем → дашборд відкривається нормально з даними.
- Викликати `POST …/webhook/jqka-run` без авторизації → відхилено (401/403), ручний запуск трекінгу НЕ стартує.
- Перевірити налаштування Webhook-вузлів у n8n (authentication: Basic Auth/header-auth), а не лише перевірку параметра в коді, яку легко обійти.
- Перевірити стан «TEMP Sheets Proxy (Claude)» у n8n → воркфлоу деактивовано (виклик `claude-sheets-proxy` повертає помилку/не знайдено).
- Перевірити чек-лист безпеки → містить інструкцію перевипуску ключа Maersk на developer.maersk.com і крок внесення нового ключа в n8n Credentials користувачкою; є відмітка, що ключ дійсно перевипущено (дата).
- Скан репозиторію на секрети (див. наскрізний ТЕСТ БЕЗПЕКИ ДОСТУПІВ) → застосувати і до нових файлів цієї задачі.

### Задача 1.9 — Інструкція credentials ліній (без секретів у репо)
- Перевірити наявність `dispatch/docs/CREDENTIALS.md`.
- Прочитати документ → описано конкретно: як зайти в n8n → Credentials → яку назву credential створити для MSC-кабінету і CMA-кабінету, які поля (логін/пароль), як воркфлоу посилаються на них за іменем/ID (а не хардкод у коді вузла).
- Скан репозиторію (git grep за патернами password/пароль/pass/pwd/логін:пароль, довгі base64/hex рядки що виглядають як ключі) → 0 знайдених реальних значень секретів у CREDENTIALS.md і будь-де в repo.
- Переглянути JSON MSC/CMA-воркфлоу → авторизація підключена через `credentials: {..., "name"/"id": ...}` вузла, а не через query-параметри/заголовки з вписаним логіном-паролем у самому JSON.

### Задача 2.2 — Дедлайни: підсвітка + авто-нагадування
- Прочитати документацію порогів → конкретні числові межі жовтий/червоний для кожного типу дедлайну (ETA-7 імпорт, Т1, демередж, зберігання, постановка авто, детеншен; для експорту — Т1, Port Cut Off, SI cut-off, детеншен, зберігання).
- Тестовий імпортний рядок з ETA через 7 днів → «Нагадування» отримує текст «запитати документи у клієнта»/«надати документи клієнту» відповідно; підсвітка кольору за порогом.
- Тестовий рядок за межею порогу (наприклад 2 дні до дедлайну) → колір підсвітки червоний (за задокументованою межею), не жовтий.
- Тестовий рядок з демереджем → перевірити, що кількість вільних днів береться з довідника «Демередж» (3.2) по комбінації Лінія×Порт×Вивіз×Обладнання, а не з хардкод-числа в коді, окремо для Maersk/MSC/CMA.
- Тестовий експортний рядок з наближенням Port Cut Off/SI cut-off → відповідний текст у «Нагадування».
- Тестовий рядок зі статусом «Вантаж доставлено» (заморожений) → підсвітка дедлайнів/демереджу НЕ спрацьовує (аналогічно поточній логіці `demRisk()`, яка повертає порожнє для доставлених).

### Задача 2.3 — Операційна дошка: фільтр «мої»
- Відкрити подання з фільтром «Операційний менеджер» = конкретне ім'я → видно лише рядки, де колонка «Оп.менеджер» дорівнює цьому імені; кількість рядків збігається з ручним підрахунком фільтром у самій таблиці.
- Змінити фільтр на інше ім'я → список рядків змінюється відповідно новому імені.
- Фільтр «Всі»/порожній → показані всі рядки без винятку.

### Задача 2.4 — Керівнича дошка: фільтри + лічильники
- Прочитати `dashboard.code.js` → присутні фільтри Менеджер/Оп.менеджер/Клієнт/Лінія з дефолтним значенням «Всі».
- Застосувати фільтр Лінія=Maersk → таблиця показує тільки рядки з Лінія=Maersk; усі лічильники (Всього/У дорозі/Прибули в порт/Доставлено/Зміни ETA/Демередж-ризик) перераховані ТІЛЬКИ по цій вибірці (звірити вручну підрахованими числами по відфільтрованих рядках).
- Застосувати два фільтри одночасно (Клієнт=X + Лінія=Maersk) → вибірка є перетином умов, а не об'єднанням.
- Застосувати фільтр із порожньою вибіркою (комбінація, що не дає жодного рядка) → сторінка не падає з помилкою, показує порожню таблицю/0 у лічильниках.

### Задача 2.5 — Клієнтська вітрина: без цін, точки маршруту, розділення

- Тестовий рядок Напрямок=Імпорт vs Напрямок=Експорт → рендер точок маршруту різний і відповідає напрямку (перевірити код розрізняє через `isExp()`/аналог, а не показує однаковий набір точок завжди).

### Задача 3.1 — Модель ролей і розділення даних (doc)
- Перевірити наявність `dispatch/docs/ROLES-ARCHITECTURE.md`.
- Прочитати документ → описано усі 4 ролі (клієнт/менеджер/оп.менеджер/керівник) і як кожна з 3 дошок (Операційна/Керівнича/Клієнтська) мапиться на ролі.

- Прочитати розділ про межу «зараз/після сайту» → пояснено, чому токен-вітрина (`?t=token`) не є строгою безпекою (токен можна переслати, немає expiry/revocation, немає per-сесійної автентифікації) і що саме дає перехід на логін (відкликання доступу, аудит по користувачу, відсутність ризику пересилання посилання).

### 1. ТЕСТ «скрипт не перетер ручні правки»
**Прив'язка: 1.2, 1.5 (головна), 1.6, 1.7, 2.1**
- Дія: у тестовому рядку вручну вписати в трековану колонку (Статус, ETA план/факт, Судно, Контейнер) значення, відмінне від того, яке видасть скрипт на цьому прогоні → прогнати оновлення (Maersk/MSC/CMA — кожну лінію окремо, коли вона реалізована).
- Результат: ручна правка лишається на місці в комірці (НЕ затерта); у «Нагадування» з'явився запис розбіжності (яке значення пропонує скрипт vs що стоїть вручну, дата). Повторити для кожної трекованої колонки окремо і для кожної реалізованої лінії.

### 2. ТЕСТ ЗБОЮ ДЖЕРЕЛА
**Прив'язка: 1.1 (Maersk), 1.6 (MSC), 1.7 (CMA), 1.4 (лог)**
- Дія: змоделювати по черзі 4 сценарії — (а) сайт/API лінії недоступний (timeout/5xx), (б) сесія кабінету злетіла (логін MSC/CMA невалідний), (в) сторінка/відповідь несподіваного формату (HTML замість даних, DataDome challenge, відсутні очікувані поля), (г) для Maersk окремо — порожній/протухлий токен (401 на все).
- Результат: таблиця ціла (кількість рядків і значення інших колонок не пошкоджені); постраждалі рядки позначені «потребує перевірки: <причина> <дата>»; є відповідний запис у лог-аркуш; n8n-execution завершується нормально (не лишається «running», не робить негайних повторних запитів по колу) — перевірити в execution history/логах, що після збою НЕ йде серія миттєвих повторних викликів до того самого джерела.

### 3. ТЕСТ БЕЗПЕКИ ДОСТУПІВ
**Прив'язка: 1.8, 1.9 (головні), а також будь-яка задача, що торкається credentials/webhooks/логів**
- Дія: пошук (git grep/повнотекстовий) по всьому репозиторію (усі файли `dispatch/`, включно з JSON-воркфлоу), по всіх аркушах Google-таблиці (включно з «Журнал автооновлень» і будь-якими лог-аркушами), і, де доступно, по n8n execution logs — за патернами: пароль/password/pass/pwd, логін:пароль комбінації, довгі рядки, що виглядають як API-ключі/токени, значення в query-параметрах webhook.
- Результат: 0 знайдених реальних паролів/токенів у відкритому вигляді будь-де з перерахованого. Посилання на credentials — лише за назвою/ID n8n-об'єкта, ніколи саме значення секрету.

### 4. ТЕСТ РОЗДІЛЕННЯ КЛІЄНТІВ
**Прив'язка: 2.5 (головна), 3.1**
- Дія: для щонайменше 3 різних клієнтів (обов'язково включити клієнта з аліасом — напр. Мірандор=Олео Захід) відкрити клієнтську вітрину за їхніми токенами по черзі; додатково перевірити крайні випадки:
  а) клієнт з аліасом — токен показує рядки під обома іменами (основне + аліас), і нічого зайвого;
  б) клієнти зі схожими/частково збіжними назвами (напр. РТК=Магнохром і будь-який інший клієнт із подібним написанням) — перевірити точний (exact-match), а не частковий/substring збіг імені;
  в) чужий токен — підставити токен, що належить іншому клієнту, у власне посилання — показуються лише дані власника цього токена;
  г) порожній/невалідний/неіснуючий токен — сторінка показує «Невірне або застаріле посилання» (або аналог) і не показує жодного реального рядка з даними.
- Результат: на дошці кожного клієнта немає жодного рядка з колонкою «Клієнт», що не належить (з урахуванням аліасів) саме цьому клієнту; крайні випадки (а)-(г) усі проходять без витоку чужих даних.

### 5. КОНТРОЛЬНА ЗВІРКА (після КОЖНОЇ автоматизації лінії)
**Прив'язка: 1.2/1.3/1.4 (Maersk), 1.6 (MSC), 1.7 (CMA) — виконувати окремо після впровадження/зміни кожної лінії**
- Дія: обрати випадкову вибірку N=3–5 (не більше 5, відповідно до обмеження обсягу ≤5 контейнерів на MSC/CMA) активних (не заморожених статусом «Вантаж доставлено») коносаментів/контейнерів цієї лінії. Відкрити офіційний публічний трекінг лінії вручну в браузері (Maersk track-and-trace / MSC / CMA еCargo) і звірити з даними в таблиці:
  - дата ETA (план і факт) — точний збіг дати;
  - назва судна і номер вояжу — збіг, коли поле в таблиці заповнене (враховуючи, що воно заповнюється лише коли днів до ETA ≤7);
  - статус (за новою 8-моделлю) — відповідає реальній стадії вантажу з трекінгу лінії (напр. «Discharged» на сайті → «Вивантажений в порту прибуття» в таблиці);
  - номер контейнера — збігається з «Контейнер (лінія)»/немає непояснених розбіжностей у «Звірка».
- Результат: усі N обраних записів збігаються з ручною звіркою без непояснених розбіжностей (>1 дня різниці в даті чи невідповідний статус). Якщо хоч один запис розходиться без пояснення — задача НЕ приймається, потрібно розібратись у мапінгу/коді перед повторною валідацією.
