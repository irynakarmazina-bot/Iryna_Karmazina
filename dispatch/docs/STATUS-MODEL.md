# Модель статусів

## Канонічна послідовність (нова 8-модель, задача 1.3, узгоджено 06.07.2026)

1. **Букінг** — стартовий/дефолтний статус до появи будь-яких подій трекінгу (вручну або з листа Експедитора). Maersk-мапінг цей статус НІКОЛИ не виставляє і не перезаписує — тільки початкове значення рядка.
2. **Стафіровка** — контейнер виїхав з депо (порожній) для завантаження вантажем у відправника, до прибуття у порт відправлення.
3. **В порту відправлення** — повний контейнер заведений на термінал порту відправлення, очікує завантаження на судно.
4. **Завантажений на судно** — контейнер завантажено на борт у порту відправлення.
5. **В морі** — від відходу судна до вивантаження в порту прибуття (включно з проміжними перевалками/трансшипментом).
6. **Вивантажений в порту прибуття** — контейнер вивантажено з судна в порту прибуття.
7. **Завантажений на авто/потяг** — вивезення контейнера з порту прибуття вглиб (авто або залізницею) — ОДИН об'єднаний статус (без розділення на «потяг»/«авто»).
8. **Вантаж доставлено** — фінальний статус: вантаж видано отримувачу і порожній контейнер повернуто. **ЗАМОРОЖЕНО** для імпорту (Maersk-мапінг більше не перезаписує).

### Що прибрано порівняно зі старою моделлю
- «На кордоні» — прибрано як окрема стадія (не було надійного джерела події саме на кордоні).
- «Зданий в порт» — об'єднано з «В порту відправлення».
- «Прибув в сухій порт» — об'єднано з «Завантажений на авто/потяг» (обидві RAIL/TRUCK-стадії після вивантаження в порту прибуття тепер один статус).
- «Доставка отримувачу» — об'єднано з «Завантажений на авто/потяг» (проміжна стадія «привезли, ще не повернули порожній» більше не виділяється окремо; фінал — лише «Вантаж доставлено»).
- «Завантажений на потяг» / «Завантажений на авто» як ДВА окремих статуси — об'єднано в один «Завантажений на авто/потяг».

### Заморозка (не перезаписувати)
- **Імпорт**: статус = «Вантаж доставлено» → Розбір більше НЕ чіпає поле «Статус» цього рядка (жодна нова подія Maersk його не змінить).
- **Експорт**: статус = «Вивантажений в порту прибуття» → це фінальна стадія для експортних відправлень (далі вантаж вже поза нашим наглядом), поле «Статус» так само заморожено.
- Заморозка стосується ЛИШЕ колонки «Статус». ETA/Судно/Контейнер по такому рядку можуть і далі оновлюватись (наприклад технічне уточнення дати), якщо для цього є нові дані і немає ручного конфлікту (guard 1.5).

## Maersk-мапінг (rozbir.code.js) — за кодом+модою ОСТАННЬОЇ (найпізнішої за eventDateTime) події

Джерело кодів: `equipmentEventTypeCode` / `transportEventTypeCode` / `shipmentEventTypeCode` (перше непорожнє); мода — `transportCall.modeOfTransport`. Порожня мода трактується як VESSEL-контекст (`isVesCtx`), бо в первинних/морських подіях мода іноді не вказана.

| Код події | Мода (`modeOfTransport`) | → Статус (нова 8-модель) |
|---|---|---|
| GTOT | VESSEL / (порожньо) | Стафіровка |
| GTIN | VESSEL / (порожньо) | В порту відправлення |
| LOAD | VESSEL / (порожньо) | Завантажений на судно |
| DEPA | VESSEL / (порожньо) | В морі |
| ARRI | VESSEL / (порожньо) | В морі (проміжна перевалка, доки не буде DISC) |
| DISC | VESSEL / (порожньо) | Вивантажений в порту прибуття |
| LOAD / DEPA / ARRI / GTOT / GTIN | RAIL / TRUCK | Завантажений на авто/потяг |
| **GTIN** | **TRUCK** | **Вантаж доставлено** (виняток із правила вище — див. нижче) |

**Порядок перевірки важливий**: спочатку перевіряється виняток `GTIN+TRUCK` (фінал), потім загальне правило «RAIL/TRUCK → Завантажений на авто/потяг» (охоплює LOAD/DEPA/ARRI/GTOT/GTIN), і лише потім — VESSEL-специфічні правила. Це покриває всі 6 кодів × {VESSEL, RAIL, TRUCK, порожньо} = 24 комбінації. Дві комбінації (`DISC`+RAIL, `DISC`+TRUCK) не мають природного відповідника в 8-моделі — рідкісний/нетиповий випадок; вони не мовчать: рядок отримує «потребує перевірки: незнайома подія DISC/RAIL(або TRUCK) <дата>» у «Нагадування» + запис у «Журнал автооновлень» (Колонка=«Статус»), а не тиху відсутність оновлення (вимога задачі 1.3 — жодної «дірки» без запису).

**Чому GTIN/TRUCK = доставлено:** у подіях Maersk поле `emptyIndicator` НЕ заповнюється, тож повернення порожнього визначаємо за останньою подією GTIN на авто (цикл GTOT→GTIN→GTOT→GTIN у кінці = видача вантажу + повернення порожнього).

## Заморозка в коді
`meta.napryamok==='імпорт' && meta.curStatus==='Вантаж доставлено'` АБО `meta.napryamok==='експорт' && meta.curStatus==='Вивантажений в порту прибуття'` → блок мапінгу статусу в `rozbir.code.js` пропускається повністю (жодного розрахунку `st`, жодного запису, жодного «Нагадування» по цьому полю).

## Guard (задача 1.5) для статусу
Перед записом нового статусу порівнюємо ПОТОЧНЕ значення комірки «Статус» з останнім значенням, яке сам скрипт туди писав (з «Журналу автооновлень», задача 1.4). Якщо вони різняться — вважаємо це ручною правкою людини, НЕ перезаписуємо «Статус», а додаємо в «Нагадування» розбіжність «трекінг: `<нове>` ≠ ваше `<поточне>`, перевірте `<дата>`». Детальніше — коментарі на початку `rozbir.code.js`.

## Ключ трекінгу (задача 1.2, скорочено — повністю в vidbir.code.js)
- BL = 9 цифр, Лінія=Maersk → трекаємо по BL, «Ключ трекінгу»=«BL».
- Немає BL, є «Контейнер», Лінія=Maersk → трекаємо по контейнеру (параметр `equipmentReference` — TODO підтвердити на живому API), «Ключ трекінгу»=«контейнер (тимчасово)».
- Є «Контейнер», Лінія порожня → «Ключ трекінгу»=«вкажіть лінію», без запиту.
- Немає ні BL, ні контейнера → «Ключ трекінгу»=«не трекається», без запиту.
- Автоперехід: якщо рядок трекався по контейнеру і з'явився BL — наступний прогін переходить на BL-трекінг, запис «контейнер → BL, `<дата>`» у «Зміни ETA (історія)» + «Журнал автооновлень».
- Матч-ключ запису в CRM — колонка «Угода» (єдина для всіх гілок; обґрунтування — коментар на початку `vidbir.code.js`), а НЕ «BL/Booking»/«Контейнер».

## ETA / судно (той самий вузол)
- ETA порт: з останньої події ARRI. `eventClassifierCode=ACT` → «ETA порт (факт)»; `EST` → «ETA порт (план)» + лог зміни в «Зміни ETA (історія)» і «Журнал автооновлень».
- Судно/Вояж: з останньої VESSEL-події, лише якщо днів до ETA ≤ 7.
- Контейнер (лінія): усі `equipmentReference` з подій; розбіжність із «Контейнер» → прапорець у «Звірка».
- «Контейнер» саму комірку скрипт пише лише якщо вона порожня (fill-if-empty, задача 1.5) — якщо людина вже вписала своє значення, скрипт її не чіпає.

## Видимість збоїв (задача 1.1, скорочено — повністю в rozbir.code.js)
- 429 → до 3 повторів САМЕ ЦЬОГО запиту з паузами 2с/4с/8с у межах прогону; після вичерпання — «потребує перевірки: ліміт запитів `<дата>`» + лог, решта батчу штатно.
- 401 (на весь запуск подій) → прогін зупиняється явно (`Стоп: 401 (events)` / `Стоп: недійсний токен`), один запис у лог, жоден рядок НЕ позначається «оновлено».
- Порожній `events`/404 → тільки цей рядок: «потребує перевірки: BL не знайдено `<дата>`» + лог.
- HTML/нема `events`/timeout → «потребує перевірки: формат/timeout `<дата>`» + лог; воркфлоу завершується, не зависає.
